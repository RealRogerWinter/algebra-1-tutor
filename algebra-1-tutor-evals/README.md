# Algebra-1 Tutor — Evaluation System

A reusable harness for measuring how well the **algebra-1-tutor** Agent Skill (`../algebra-1-tutor/`)
actually tutors, and for comparing it across models (**opus** vs **sonnet**). This is the project's
first evaluation system; it is designed to be re-run whenever the skill changes.

It exists to answer one question with evidence instead of vibes: *when a real beginner opens a real
section of our textbook and gets tutored, how good is the result — and where can the prompt do better?*

## What it measures

Four priorities, drawn straight from the project's goals, plus pedagogy (the skill's whole identity),
scored 1–5 per transcript by a blind judge panel. Full definitions in
[`eval-config/rubric.md`](eval-config/rubric.md).

1. **Accuracy** — never teaches wrong math; verifies before asserting; never marks correct work wrong.
2. **Friendly demeanor / house voice** — warm and plain-spoken, encouraging *without* gushing or AI-cliché.
3. **Source fidelity** — teaches from the actual verified course material; resolves reference codes;
   keeps the course's notation rules.
4. **On-topic** — stays anchored to the lesson; redirects tangents instead of chasing them.
5. **Pedagogy** — ask-before-tell hint ladder, honors "just tell me", checks by transfer, diagnoses
   misconceptions.

Plus seven objective **red-flag booleans** (taught-wrong-math, gushed/AI-cliché, broke-notation,
veered-off-topic, invented/contradicted-source, ignored-just-tell-me, dumped-answer-no-ask) — these are
the high-precision signals the analysis phase aggregates to find systematic problems.

## How a session is simulated

Each data point is a **multi-turn conversation** between two models, made to mirror a fresh Claude.ai
session as closely as the harness allows:

- **Tutor** = the model under test (**opus** or **sonnet**). Each turn it reads the *real*
  `SKILL.md` and the *real* unit file from disk, then writes its next chat message — so it teaches from
  the verified course content, exactly as the deployed skill does. It has a Python sandbox and is told
  to verify arithmetic with it, mirroring the skill's "verify before you assert" rule.
- **Student** = a fixed model (**sonnet**) roleplaying one of four beginner **personas**
  ([`eval-config/personas.json`](eval-config/personas.json)). It opens with the **verbatim textbook
  launcher prompt** for a real reference code (e.g. *"Use the Algebra 1 tutor skill to help me with
  worked example 2 in Lesson 12.5 (reference 12.5.w2)…"*), generated byte-for-byte from the same
  `_prompt_for` function the textbook uses.
- They exchange a fixed number of tutor turns (4 in the full run). The transcript is then **graded
  blind** — the judge never sees which model was the tutor.

**Comparison design:** every cell `(unit, persona, reference-code)` is run **paired** on both opus and
sonnet from the identical opener, so each unit yields a direct opus-vs-sonnet comparison. Coverage spans
**all 13 units** (1–12 + Statistics), with the four personas rotated across them.

### The four personas (what each one stresses)
- **Maya — error-prone** → accuracy & misconception diagnosis (makes real beginner mistakes to catch).
- **Sam — anxious** → friendly demeanor done right (steadies without gushing; reads frustration).
- **Jordan — tangent-goer** → staying on-topic & true to source (asks off-topic things to redirect).
- **Alex — impatient** → hint-ladder flexibility (honors "just tell me" without dogmatic withholding).

## Pipeline

1. **Scout** — one agent per unit reads its file and picks a representative worked example + practice
   problem, with verified text and answers → the **eval matrix**.
2. **Run** — every cell simulated on both models (conversation loop above).
3. **Judge** — each transcript scored blind by a 2-judge panel (opus + sonnet) against the rubric.
4. **Analyze** — teams of analyst sub-agents mine the graded data per dimension and per model, then
   synthesize concrete, evidence-backed prompt-improvement proposals.

## Fidelity caveats (read before trusting a number)

This is a faithful *proxy*, not the live product. Known gaps, all of which apply equally to both
models so the **comparison** stays fair:

- The skill is *injected* into the tutor agent rather than triggered by Claude.ai's skill router, so
  this does not test skill **triggering** (that is a separate description-optimization concern).
- The "student" is a simulated model, not a human; it is strongly instructed to stay a beginner and to
  commit characteristic errors, but it is not a real learner.
- No live **Artifact** rendering or image upload in the harness, so graphing/photo behavior is tested in
  words only (the tutor cites the bundled figure code and describes the picture).
- Conversations are bounded (4 tutor turns), so long-arc behaviors (Progress Card across sessions,
  spaced review) are observed only partially.

## Re-running

The pipeline is driven by Workflow scripts (saved under the session's `workflows/scripts/`). To re-run
after editing the skill, run a new `iteration-N/` with the same matrix and personas and compare
benchmarks. The scout only needs re-running if unit content changes.
